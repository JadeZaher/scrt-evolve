# scrt-evolve — Design

> **Status: design draft — for review before any implementation.** No code
> yet. This document fixes the crate/module shape, the core trait
> boundaries, the config schema, the CLI + SDK surface, and the honest
> risks. Nothing here is built; review and revise this first.

## What it is

**scrt-evolve** is a standalone Rust framework that makes a model *better at
its own corpus* with no human labeling. You give it three things:

1. a **model path** (a local model directory — safetensors weights + tokenizer),
2. a **corpus directory** (the body of work to adapt to), and
3. **mind-palace data** (a scrt palace — the curated retrieval signal),

and it runs a self-contained loop:

```
   discover            generate                 train
 ┌──────────┐      ┌───────────────┐      ┌──────────────────┐
 │ use scrt │  ->  │ synth QA /     │  ->  │ finetune via a   │
 │ to find  │      │ finetuning     │      │ preset (lora /   │
 │ context  │      │ pairs from it  │      │ full / pretrain /│
 │ from the │      │ (local model   │      │ contrastive /    │
 │ corpus + │      │  OR API turns) │      │ shard)           │
 │ palace   │      └───────────────┘      └──────────────────┘
 └──────────┘
```

The corpus + palace **are** the training signal: scrt discovers relevant
context, the framework turns that context into supervised data by
generating its own QA / instruction pairs, and a training preset finetunes
the model on it. It is **self-supervised in the labeling sense** — the
"labels" are model-generated from discovered context, not hand-written.

It **consumes scrt** (this workspace's retrieval engine) as a dependency —
scrt does the retrieval; scrt-evolve does the generation + training.

## Non-goals (v1)

- **Not a serving/inference stack.** It produces adapters/weights; running
  them is the host's job.
- **Not a general training framework.** It's scoped to the
  discover→generate→train loop over a corpus+palace.
- **No human-in-the-loop labeling.** Data is model-generated by design.
- **Not a distributed-systems product.** Decentralized shard training (see
  §Training presets) is a real preset but the v1 bar is "works across a
  small, trusted set of nodes," not Byzantine fault tolerance.

## Amendment 2026-06-20 — Python-first training (validated)

**Status:** End-to-end validation complete. The direction-of-record reflects this amendment.

### Candle paths are fixture-only

The candle `train`/`local` backends in the Rust crate are **mechanical/fixture paths** that cannot load real pretrained models in production. Empirically validated:
- Real models (TinyLlama-1.1B, Llama) use **RoPE** positional embeddings (no `embed_positions` tensor); our candle `model.rs` requires `embed_positions` and cannot load them.
- Real models use **GQA** (grouped-query attention) with separate KV head counts; our candle model hardcodes [2048, 2048] dense projection shapes and cannot handle [256, 2048] GQA reductions.
- Real model weights are **BF16**; our candle code assumes F32.

These paths are suitable for **overfit-tiny-batch validation** (tracks 03/04) and serve the "works as a sandbox" goal, but are **not a real-model training path**. The north-star "Rust-native training" goal is unchanged and deferred; reimplementing PyTorch's ecosystem in candle is explicitly out of scope for now.

### The real-model path: Python backend (primary)

scrt-evolve's **PRIMARY real-model training and inference path** is Python, driven from the Rust CLI via subprocess over the `dataset.jsonl` contract:

- **`python/scrt_evolve_train/`** — standalone transformers-based LoRA trainer. Loads real HuggingFace causal-LM checkpoints (handles RoPE/GQA/BF16), attaches LoRA adapters to q_proj/v_proj, trains on the dataset.jsonl schema with prompt masking, saves `adapter.safetensors` (correct GQA output shapes). Ported from lexame hivemind-models with no `peft` dependency (hand-rolled `LoRALinear`).
- **`python/scrt_evolve_infer/`** — standalone inference module. Loads base + optional adapter, runs A/B comparison (base vs base+adapter), supports sampling/temperature.
- **CLI surface:** `evolve train fit --backend transformers` shells out to `python -m scrt_evolve_train`; `evolve model infer` drives inference.

**Consistency with DESIGN.md directives:** This confirms the existing lane directive (DESIGN.md lines 55–60, "Heavy ML via PyO3→transformers (lane directive)... candle is an optional later path"). The directive was authored to defer candle ecosystem maturity and endorse Python-first heavy ML. This amendment extends the directive from the self-evolve lane (10–15) to the **core training path** (tracks 02–04).

**Dependency reality:** The Python path introduces an explicit external dependency: a managed Python environment with `torch`, `transformers`, `safetensors`. No `peft` required (avoided by design — hand-rolled LoRALinear is simpler and attribute-portable). This is a deliberate, documented trade-off: heavy ML stays independent and runnable as a subprocess; the default Rust build remains ML-free (no candle, no Python linked).

### New components and track

- **Track 19 (python-train-infer):** Standalone Python trainer/inference, driven from Rust CLI via subprocess, dataset.jsonl contract, PRIMARY real-model path (candle = fixture).
- **Track 03/04 amendments:** Candle `train`/`local` are fixture/mechanical; their sign-offs stand at that bar.

> **Scope note (self-evolve lane, tracks 10–15).** The original v1 scope above
> is "one corpus, one pass." The self-evolve lane deliberately extends it toward
> the product goal — **locally tune ONE model that evolves with a user's goals
> across ALL their projects** — and revises the non-goals accordingly:
> - *Heavy ML via PyO3→transformers (lane directive).* The lane's candle-thin
>   workflows (masked LoRA, DPO, early-exit depth-cheapen, base pruning,
>   perplexity/exit-depth scoring) are driven through HF
>   `transformers`/`peft`/`trl`/`torch` via `bridge.rs` (`--features pyo3`), not
>   hand-built in candle. The `api` paths + native Rust (router, registry, txn,
>   clustering, gate) need no Python; candle is an optional later path.
> - *Shared eval harness (track 10, lane foundation).* A single `ProbeSet` +
>   `Scorer` + `StepVerdict` + executable `gate.rs`, consumed by 11/12/15 — built
>   FIRST so they stop assuming an evaluator nobody owns.
> - *Cross-project, merged.* Discovery may union multiple projects (a new
>   `corpus_dirs: Vec<PathBuf>` on `EvolveSection`) into ONE evolving model
>   (decision: single merged model). Single-`corpus_dir` stays the default;
>   cross-project is the opt-in self-evolve mode.
> - *Outcome signal beyond "it parses."* The stronger signal is **Constitutional
>   self-refinement** (track 12): a dialectic (thesis → metacognition → Jungian
>   shadow antithesis → synthesis) against an authored-base + mined-overlay
>   constitution. Stays inside the "no human labeling" non-goal — labels are
>   model-generated, now constitution-judged rather than parse-judged.
> - *Cadence.* v1 cadence is **on-demand `scrt-evolve run` sessions**. A local
>   background **daemon** is the north-star but is explicitly NOT a build track;
>   tracks 10–15 are built non-interactive + resumable so a daemon can drive them.
> - *Faster training, all paths.* A tier-1 attribution-guided **training mask**
>   (track 13) selects which layers/modules to update (honored via PyO3→`peft`
>   target-module freezing). Distinct from distributed `shard` (nodes); `full()`
>   (no mask) stays the default. Track 13 is the single attribution owner.
> - *Modular, grow-on-demand capability.* The capability topology may grow as
>   **adapter-experts** (track 14): a path-detector finds recurring paths, track
>   13's `AttributionReport` plans each expert's target layers, each becomes a
>   PyO3→`peft` LoRA expert + registry entry, and a native **router** dispatches
>   top-k. Relaxes "not a serving stack" only to an additive router; the **base
>   model stays dense and standalone** and an empty registry is byte-identical to
>   base. Adapter-experts (≈MoLE), NOT FFN-MoE or a carved sub-model.
> - *Homeostasis: shrink, detect, revert (the base case).* The lane grows safely
>   only because it can also **prune, detect regression, and roll back** (track
>   15). Every evolution step is **transactional** — checkpoint → evaluate (via
>   track 10) → keep or **rollback** — so no step (including base pruning, done
>   via PyO3→torch) is irreversible. Self-prunes (cold experts always; gated base
>   sparsity only inside the transaction); on a **catastrophic** regression it
>   auto-rolls-back, **quarantines the cause** (by `gen`-provenance), and
>   **halts**. Consequence: a daemon may ONLY auto-evolve THROUGH this
>   transactional wrapper — that is what makes unattended evolution safe.

> **Architecture note (DAG substrate + self-architecting, tracks 16–17).** The
> framework's pipeline is re-expressed as a **typed DAG** and ultimately becomes
> a generic, self-architecting training/model-building framework:
> - *Steps are typed nodes; a run is a validated DAG (track 16).* Each stage
>   (discover/plan/generate/gate/train/eval/prune/spawn/…) is a registered `Node`
>   with typed input/output ports; a `Dag` = nodes + edges, validated at build
>   time (acyclic, port types match) and serialized to `dag.json`. The current
>   linear `run()` becomes ONE generated canonical DAG — **wrap the existing
>   stage `run` fns, do not rewrite**. This generalizes the in-tree `plan/`
>   module (a planner LLM already emits `GenPlan`/`GenSpec` artifacts instead of
>   hand-authored generation config) up one level: the *whole pipeline* is
>   planned, inspectable data.
> - *The DAG is designed from intent, not hand-authored (track 17, Part 1).* The
>   QA/interview process emits an `EvolveIntent`; a planner LLM emits a validated
>   `DagSpec` + a reproducible `evolve.toml`. Config authoring becomes config
>   *generation*.
> - *The architecture distills as ARTIFACTS (track 17, Part 2) — artifact-first,
>   not mutation-first.* The architecture design is always a first-class file
>   (`DagSpec`/`dag.json`). The system **generates those artifacts and uses
>   them**: it writes a candidate `DagSpec` file, runs it exactly as a
>   hand-authored config, and keeps proven files in a reusable **library**
>   (`arch/library/`). It is **selection-first** — on a new intent it REUSES a
>   fitting library artifact and only GENERATES on a miss — so over time it
>   re-selects more and generates less. It never mutates a live in-memory graph
>   and never synthesizes *model* architecture (layers/topology/node logic);
>   "self-distillation" = accumulating a library of proven, re-selectable
>   architecture files. Rails: (1) artifacts are typed DAGs of REGISTERED nodes
>   that pass `Dag::validate()`; (2) weight-touching trial runs go **through
>   track 15** (pass → keep weights + admit artifact; regress → roll back +
>   discard; catastrophe → halt + quarantine); (3) budget-bounded. Every artifact
>   is reproducible from its file + `evolve.toml` and re-selectable from the
>   library. (New node *implementations* — vs. wiring/cfg — remain out of scope.)
> This is what relaxes the original "not a general training framework" non-goal —
> deliberately, and only behind the typed/transactional/bounded rails above.

## How it consumes scrt

scrt-evolve depends on **`scrt-core`** (the Rust crate) for retrieval —
NOT the CLI subprocess, NOT PyO3. Direct in-process calls to
`search_with_meta`, the `palace` module, and the `simhash` similarity
surface.

```toml
# Cargo.toml (interim — git dep until scrt is published)
[dependencies]
scrt-core = { git = "https://…/scrt-cli", package = "scrt-core" }
# -> later: scrt-core = "0.1"   (crates.io)
```

> **Dependency-order note.** This repo targets a *published* scrt. Until
> that release exists, the dep is pinned to a git tag/commit. Swapping to
> the crates.io version is a one-line change tracked as its own task.

What scrt-evolve uses from scrt-core:
- `search_with_meta(&SearchConfig)` — corpus discovery (the retrieval pass).
- `palace::FilePalace` / `palace::ops` — read the mind-palace stashes.
- `palace::simhash` — rank/cluster stashes to pick what's worth generating
  data about, and to dedup near-identical context.

## Crate / module layout

A small workspace: a library crate (the SDK) + a thin binary (the CLI).

```
scrt-evolve/
  Cargo.toml                 (workspace)
  crates/
    scrt-evolve/             the SDK (lib) — everything is callable here
      src/
        lib.rs               public surface: the three stages + config
        config.rs            EvolveConfig + per-stage/per-preset config (toml)
        discover.rs          corpus + palace -> DiscoveredContext (via scrt-core)
        generate/
          mod.rs             GenBackend trait + the generate() driver
          local.rs           LocalCandle backend (candle inference)
          api.rs             ApiEndpoint backend (OpenAI/Anthropic/any, configurable)
          prompts.rs         QA / instruction synthesis prompt templates
        dataset.rs           the on-disk dataset format (jsonl) + writer/reader
        train/
          mod.rs             TrainingPreset trait + the train() driver
          lora.rs            PEFT / LoRA preset
          full.rs            full finetune preset
          pretrain.rs        continued pretraining preset
          contrastive.rs     InfoNCE embedding-adapter preset (the existing seam)
          shard.rs           decentralized shard-training preset (coordinator+worker)
        model.rs             model loading (safetensors + tokenizer) — per-arch seam
    scrt-evolve-cli/         the binary `scrt-evolve`
      src/main.rs            arg parsing -> SDK calls (discover|generate|train|run)
```

The **SDK is primary**; the CLI is a thin argv→SDK shim. Anything the CLI
does, a library consumer can do by calling the same functions — and the CLI
holds **no orchestration of its own** (the `Run` pipeline lives in the SDK as the
canonical DAG, not inline in `main.rs`). This applies to the whole surface,
including the self-evolve + architecture lanes (tracks 10–17): every node,
preset, eval, and architecture op is a public SDK item.

The **primary SDK interface is a trait-powered builder** (track 18): you design a
pipeline and then `.execute()` it. **Capability traits** select which step sets,
tag types, formats, and training tooling the builder exposes; steps are
**two-phase callbacks** (`resolve_args` → `execute`) whose phase boundary is a
**sandbox seam**. The builder **lowers to the serializable typed DAG** (track 16)
— which is the persistence format (`dag.json`) and the lowering target, not the
headline API. The original three functions remain as convenience wrappers.

```rust
// PRIMARY: trait-powered builder, designed then executed.
// The capability trait(s) determine which .step()s/formats exist (typestate).
let outs = Builder::<CoreEvolve + SelfEvolve>::new(&cfg)
    .discover()                          // each step = two phases:
    .generate(|ctx| plan_specs(ctx))     //   resolve_args(ctx) -> Args   (gen)
    .train()                             //   execute(args)     -> Out    (exec)
    .build()?                            // lowers to a typed DAG + validates
    .execute(&ctx)?;                     // runs; weight-touching exec via track 15

// the lowering target / persistence (track 16) — also runnable directly
scrt_evolve::Dag::canonical(&cfg).run(&ctx)?;
scrt_evolve::Dag::load("dag.json")?.run(&ctx)?;   // a generated/saved artifact

// convenience wrappers (unchanged) — thin calls over the canonical DAG
let ctx     = scrt_evolve::discover::run(&cfg)?;
let dataset = scrt_evolve::generate::run(&cfg, &ctx)?;
let report  = scrt_evolve::train::run(&cfg, &dataset)?;
```

## The three core traits

The whole framework is three swappable boundaries. Get these right and the
presets/backends are independent plug-ins.

### 1. Retrieval is just scrt (no trait needed)

Discovery is concrete — it calls scrt-core directly. `discover.rs` produces:

```rust
pub struct DiscoveredContext {
    /// Retrieved context chunks worth generating data about, with provenance.
    pub passages: Vec<Passage>,        // { text, source, score }
    /// Palace stashes that seeded/anchored the discovery.
    pub anchors: Vec<StashRef>,
}
```

Discovery strategy (config-driven): walk the palace stashes as seed queries
→ scrt-search the corpus → dedup via `simhash` → rank → cap by a token/passage
budget. Optionally cluster stashes by similarity so generation covers
distinct topics rather than re-mining one.

### 2. `GenBackend` — where the synthetic data comes from

```rust
pub trait GenBackend {
    /// Turn one context passage into N supervised examples (QA / instruction
    /// pairs, or raw completions). The prompt templates live in prompts.rs;
    /// the backend just executes the generation turn(s).
    fn generate(&self, ctx: &GenContext) -> anyhow::Result<Vec<GenExample>>;
}
```

Two impls, picked per run via config:

- **`LocalCandle`** — loads the user's model (the same `model_path` used for
  training) and runs candle inference to synthesize pairs. Fully offline,
  no external dependency. *Harder*: candle text-generation + sampling/decoding.
- **`ApiEndpoint`** — "turns with an API endpoint of choice": configurable
  base URL + auth + model name (OpenAI / Anthropic / OpenAI-compatible).
  Cheap to build, high-quality pairs from a strong model. Supports
  multi-turn generation (refine/critique loops).

Both produce the same `GenExample`, so the dataset is backend-agnostic.

### 3. `TrainingPreset` — how the model is updated

```rust
pub trait TrainingPreset {
    type Config;                       // preset-specific (lora rank, lr, …)
    fn train(&self, model: &LoadedModel, data: &Dataset, cfg: &Self::Config)
        -> anyhow::Result<TrainReport>;
}
```

Five presets, each with its own config block (see §Config):

| Preset | What it does | Artifact | v1 bar |
| :--- | :--- | :--- | :--- |
| **`lora`** | PEFT LoRA adapters on attn/MLP projections | `adapter.safetensors` | **primary path among candle presets*** |
| **`full`** | update all weights | full weights | works, memory-heavy |
| **`pretrain`** | continued causal-LM pretraining on the **raw** corpus (not QA pairs) — domain adaptation | weights/adapter | works |
| **`contrastive`** | InfoNCE embedding adapter from palace structure (stash note=query, nodes=positives) — improves **scrt's own retrieval** | embed adapter | the existing seam |
| **`shard`** | decentralized training split across nodes (coordinator + workers, gradient/param exchange) | merged weights | **small trusted cluster only** |

The `pretrain` and `contrastive` presets consume different inputs than the
instruction presets (`pretrain` = raw corpus passages; `contrastive` =
palace structure, not generated QA) — the `train()` driver routes the right
dataset shape to each.

\* **Note (Amendment 2026-06-20):** The candle `lora` preset is primary among the **candle-based** presets; the real-model training path that works with production checkpoints (RoPE/GQA/BF16) is Python/transformers, driven via `evolve train fit --backend transformers` (track 19). See §Amendment 2026-06-20 for details.

## Dataset format (the generate↔train boundary)

One JSONL file is the contract between stages, so generate and train are
independently runnable and inspectable:

```jsonl
{"kind":"qa","prompt":"…","completion":"…","source":"src/auth.rs","gen":"api:gpt-…"}
{"kind":"instruction","instruction":"…","input":"…","output":"…","source":"…"}
{"kind":"completion","text":"…raw passage for pretrain…","source":"…"}
{"kind":"contrastive","query":"…","positive":"…","negatives":["…"],"stash":"auth"}
```

`kind` selects which presets can consume the row. The dataset is the
durable artifact — you can generate once and train many presets from it,
or hand-inspect/edit it before training.

## Config schema

One `evolve.toml`, with a top section + per-stage + per-preset blocks. Every
stage reads only what it needs, so partial configs work (generate-only,
train-only).

```toml
[evolve]
model_path  = "/models/my-model"     # the ONE required thing (weights + tokenizer)
corpus_dir  = "./src"                # what to adapt to
palace_path = ".mpg/mind-palace.json"# the retrieval signal
work_dir    = ".scrt-evolve"         # datasets, adapters, checkpoints land here

[discover]
seed = "palace"            # palace | corpus | both — where discovery starts
max_passages = 500
dedup = "simhash"          # use scrt's similarity to drop near-dup context
cluster = true             # spread generation across distinct topics

[generate]
backend = "api"            # local | api
kinds = ["qa", "instruction"]
per_passage = 3            # examples synthesized per passage
  [generate.local]
  max_new_tokens = 512
  temperature = 0.7
  [generate.api]
  base_url = "https://api.…/v1"
  model = "…"
  api_key_env = "SCRT_EVOLVE_API_KEY"   # never inline secrets
  turns = 1                              # multi-turn refine if >1

[train]
preset = "lora"            # lora | full | pretrain | contrastive | shard
  [train.lora]
  rank = 16
  alpha = 32
  target_modules = ["q_proj","v_proj"]
  lr = 2e-4
  epochs = 1
  [train.full]
  lr = 1e-5
  epochs = 1
  grad_accum = 8
  [train.pretrain]
  lr = 1e-5
  block_size = 1024
  [train.contrastive]
  negatives_per_row = 4
  temperature = 0.05
  [train.shard]
  role = "coordinator"     # coordinator | worker
  peers = ["host:port", …]
  shard_strategy = "data"  # data | layer
  base_preset = "lora"     # what each shard runs locally
```

`api_key_env` (not a literal key) is the only safe way to pass secrets;
the framework reads the named env var.

## CLI surface (thin shim over the SDK)

```bash
# full pipeline in one shot
evolve train run --config evolve.toml

# or each stage independently (the composable form you described)
evolve train discover --config evolve.toml            -> work_dir/discovered.json
evolve train generate --config evolve.toml [--in discovered.json]  -> dataset.jsonl
evolve train fit      --config evolve.toml [--data dataset.jsonl]  -> adapter/weights

# presets are selectable inline too (override config)
evolve train fit --preset lora --data dataset.jsonl
evolve train fit --preset contrastive   # consumes palace structure directly
evolve train generate --backend api     # override the configured backend
```

Each stage reads/writes the work-dir artifacts, so you can stop, inspect,
edit, and resume between stages — `generate` consumes `discover`'s output,
`train` consumes `generate`'s dataset. This is the "dataset generation
exposed in a similar way, consumed by the training CLI calls" you asked for.

## SDK surface (the same, as a library)

```rust
use scrt_evolve::{EvolveConfig, discover, generate, train};

let cfg = EvolveConfig::load("evolve.toml")?;
let ctx     = discover::run(&cfg)?;                 // -> DiscoveredContext
let dataset = generate::run(&cfg, &ctx)?;           // -> Dataset (jsonl-backed)
let report  = train::run(&cfg, &dataset)?;          // -> TrainReport

// or drive a single stage / swap a backend programmatically
let backend = generate::ApiEndpoint::from_cfg(&cfg.generate)?;
let examples = backend.generate(&gen_ctx)?;
```

## Build order (proposed phases, each review-gated)

1. **Repo skeleton + config** — workspace, `EvolveConfig` (toml load +
   validate), work-dir layout. No ML. *(compiles, `config` tested)*
2. **Discover** — `discover.rs` consuming scrt-core (search + palace +
   simhash dedup/cluster) → `DiscoveredContext`. *(no ML; tested against a
   fixture palace)*
3. **Generate / API backend** — `ApiEndpoint` + prompt templates +
   dataset writer. End-to-end discover→dataset with no local model.
4. **Generate / local backend** — `LocalCandle` inference behind the same
   trait (the harder candle piece).
5. **Train / LoRA** — the primary preset: model loader + LoRA injection +
   training loop → `adapter.safetensors`.
6. **Train / contrastive** — port the existing InfoNCE seam from this
   workspace's `scrt-evolve` crate.
7. **Train / full + pretrain** — the two heavier generative presets.
8. **Train / shard** — decentralized coordinator+worker. Last, and scoped
   to a small trusted cluster.
9. **Extract / publish** — swap the scrt git dep for the published crate;
   cut the repo's own first release.

## Honest risks (state these up front)

- **candle's finetuning ecosystem is thin.** No turnkey PEFT/trl
  equivalent. LoRA injection, the optimizer/training loop, gradient
  checkpointing, and **per-architecture model loaders** are largely
  hand-built. Each model family (Llama/Qwen/Mistral/…) needs its candle
  implementation; "load any safetensors and finetune" is aspirational, not
  free. Mitigation: start with ONE well-supported arch, make `model.rs` a
  clean seam, expand coverage as a backlog.
- **Local generation quality.** A small local model generating its own
  training data can amplify its own errors (model collapse / echo
  chamber). The API backend sidesteps this with a stronger teacher.
  Mitigation: ship API-first quality, treat local-gen output as lower-trust
  (dedup, filter, optional critique turn).
- **Decentralized shard training is genuinely hard.** Gradient/param
  exchange, stragglers, failure handling, consistency. The v1 bar is a
  small trusted cluster, not a robust distributed system. It's deliberately
  last in the build order.
- **The whole premise is unproven at quality.** "Self-generated data
  finetunes a usefully-better model" is plausible, not guaranteed — same
  honest caveat as the original spike. The pipeline being *wired and
  inspectable* (you can read the dataset, swap the teacher) is what makes
  the bet measurable rather than blind.

## Relationship to the in-tree `scrt-evolve` crate

This workspace already has a `crates/scrt-evolve` (corpus export + InfoNCE
seam behind a `train` feature). That code is the **seed** for the
`contrastive` preset and the corpus/dataset plumbing here — it gets lifted
into this repo (phase 6), and the in-tree crate can then be retired or kept
as a thin re-export. Decide at extraction time (phase 9).
